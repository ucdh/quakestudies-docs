Consistency of data

One of the gaols for Quakestudies Repository is to ensure the long-term consistency of data stored in the repository. This means ensuring that any changes to the ontology or the way data is stored in the repository are of minimal consequence to a researcher wanting to access data across a time series.

In order to promote this long-term consistency of the Quakestudies data, the following need to be borne in mind:

Changes to the ontology

The Drupal interface makes it easy for an administrator to add, modify or delete fields; however it must be understood that doing so may have major consequences for the data consistency.

  • Adding a new field via the Drupal interface will not result in it being added to any existing class instances; only new class instances will pick up this new field. If an existing class instance is edited by a user using the Drupal interface, then it will pick up this new field only if some data is entered into it. Therefore the addition of a new field will result in some instances of the class having this new field, and some (the pre-existing instances) not having this field.
  • Deleting a field via the Drupal interface will not result in any existing class instances having this field (and the data that it contains) removed; only when new class instances are added (or existing class instancres modified) via the Drupal interface will the user no longer have the ability to enter data into this field.

In order to ensure data consistency, the following is recommended:

  • rather than delete a field via the Drupal interface, it is suggested that instead the field is marked as deprecated in the ontology, and in the Drupal interface the field is hid such that a user never sees this field.
    This will require
    • that the field is marked as hidden in the Drupal interface; this will ensure that any contents of the field are hidden from the end user on the Drupal display interface
    • that the field is marked as deprecated in the Drupal edit interface
  • note that, as Fedora keeps a record of each version, it is still possible to examine any previous state of a particular Fedora record after a field is added or deleted, albeit only via the Fedora Admin interface.

Consistency of data capture

With many of the fields being free-text, the Qakestudies Superuser and Content Loaders have considerable leeway when entering data as to the way the data is stored in a given field.

It would be advantageous for data consistency if the Quakestudies administrators formulate and adhere to some house style rules regarding the capture of data, such as

  • Length of data stored in certain fields; In some cases, fields will be truncated for display via the Drupal interface
  • Nomenclature regarding naming class instances (Collection, Object, Part etc)
  • Rules regarding when and wen not to capture ancilliary data such as Party, Person etc (for example, is it worth creating an instance of the Person class if no other details besides the person's name is known?)
